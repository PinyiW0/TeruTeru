# Vibe E2E — 產生並執行 vibe spec（v1）

## 目的

把 vibe 階段新增的**互動行為**與**結構新增**自動轉成 Playwright spec，並執行驗證。

不測：純 visual、業務合約（那是 `/vibe-check` 主 spec 的工作）。

## 前置條件（軟性 gate）

執行前**應確認**：

1. **`/vibe-check` 已綠燈**——若主 spec 紅燈，vibe 改動本身可能違規，vibe spec 結果意義不大
2. 已用 `/vibe-setup` 看過分層報告（可選但建議）——確保理解這次會產哪些 spec

執行時若**無法確認**這次 session 已跑過 vibe-check，請先告知使用者並建議補跑，使用者點頭再繼續。

## 使用方式

```bash
/vibe-e2e            # 對比 HEAD：產 spec + 跑
/vibe-e2e <ref>      # 指定 base ref
/vibe-e2e --dry-run  # 只產 spec、不跑（讓使用者預覽）
```

---

## 絕對禁止

- 不可動 `test/e2e/specs/`（主 spec 凍結）
- 不可動 `spec/gherkin-feature/`、`spec/e2e-flows/`
- 不可動 `playwright.config.ts`（主 config 凍結；vibe 用 `playwright.vibe.config.ts`）
- 不可主動 commit / push
- 不可在 `test/e2e/vibe/` 之外寫入

## 流程

### Step 0：前置確認

確認這次 session 已跑過 `/vibe-check` 綠燈。若無紀錄，提示使用者「建議先跑 /vibe-check 確認主 spec 綠燈」，等使用者點頭。

### Step 1：分類 vibe diff

```bash
git diff <base> -- app/pages/ app/components/ app/layouts/
```

分類規則（grep pattern、sub-pattern mapping、邊界 case）見 [`../vibe-setup/classification.md`](../vibe-setup/classification.md)，與 `vibe-setup` 共用。

**只處理「互動」與「結構」hunk**，純 visual 略過。

### Step 1.5：載入合約 testid 白名單

```bash
# 提取主 spec 所有 getByTestId 字串作為合約白名單
SPEC_TESTIDS=$(grep -rhoE "getByTestId\(['\"]([^'\"]+)['\"]\)" test/e2e/specs/ \
  | sed -E "s/.*['\"]([^'\"]+)['\"].*/\1/" | sort -u)
```

這個集合 = vibe spec 可直接使用的 **testid 合約白名單**。之後 Step 3 套 pattern 模板時，locator 推測規則會用這個集合做檢查。

### Step 2：清空舊 vibe spec

只清前次 auto-gen 的 spec（保留人工放置的檔案、`helpers.ts`、`README.md`）：

```bash
# 用 marker 篩，未含 marker 的視為人工檔不清
grep -l 'AUTO-GENERATED by /vibe-e2e' test/e2e/vibe/*.spec.ts 2>/dev/null | xargs -r rm -f
```

理由：vibe spec 是**短命的 auto-gen 產物**，每次重生對應當下 diff。但人工放置的 vibe spec（無 marker）不在 skill 管轄範圍，不該被誤刪。如果某個情境需要長期維護的測試，應該寫到主 spec。

### Step 3：依 pattern 套版產生 spec

對每個互動 / 結構 hunk，依 sub-pattern 套對應模板（見下方「Pattern 模板表」），產出 `test/e2e/vibe/{layer}-{module}-{pattern}-{n}.spec.ts`。

每個檔案**首行加 marker**：

```ts
// AUTO-GENERATED by /vibe-e2e — do not edit; regenerate via /vibe-e2e
// Source hunk: app/pages/practice/live.vue:120-135
// Pattern: toggle
```

不確定的 locator 用 `// TODO:` + `.skip` 標記（不阻塞綠燈判定）。

### Step 4：跑 vibe spec

```bash
npx playwright test --config playwright.vibe.config.ts
```

`--dry-run` 模式跳過此步。

### Step 5：回報

**綠燈**：

```
=== Vibe E2E 通過 ===

產生 5 個 vibe spec（清空 0 個舊檔）
跑了：8 cases，8 passed ✅（含 2 .skip 為 pattern 不確定）

涵蓋：
- interaction-practice-toggle-1：sidebar 折疊 → 通過
- interaction-players-tab-1：球員 tab 切換 → 通過
- structure-players-empty-state-1：空狀態顯示 → 通過
- structure-accounts-loading-state-1：loading UI → 通過
- structure-cameras-error-state-1：error UI → 通過

vibe 改動的互動/結構驗證通過，可以 commit。
```

**紅燈**：

```
=== Vibe E2E 失敗 ===

8 cases，2 failed ❌

失敗清單：

1. interaction-practice-toggle-1.spec.ts › vibe：sidebar 折疊
   失敗訊息：點觸發按鈕後 sidebar 仍 visible
   來源 hunk：app/pages/practice/live.vue:120-135
   pattern：toggle
   嫌疑：toggle 邏輯沒接好，或目標元件 locator 推測錯誤
   建議行動：
   - 對照 app/pages/practice/live.vue 確認 ref 與 @click 綁定
   - 若 locator 推測錯誤，請手動跑 npx playwright test --config playwright.vibe.config.ts --ui 用 codegen 找正確 locator

2. ...

下一步建議：
- 修 UI 後重跑 /vibe-e2e 驗證
- 若 pattern 推測明顯不適用，請手動跑 /vibe-check 確認主 spec 仍綠，並考慮 vibe 改動是否要保留
```

**孤兒 testid 警告**（無論綠紅都要列於 Step 5 輸出末尾）：

```
⚠️ 孤兒 testid 警告：

vibe diff 中使用了 N 個不在主 spec 合約白名單的 testid，已自動 .skip：

  - app/components/Sidebar.vue:30 → 'sidebar-collapse-button'
    建議：
    (a) 改用 getByRole('button', { name: /折疊/ }) — 推薦
    (b) 若這是長期合約，請改 .flow.md → /test e2e spec → 主 spec 收進後再啟用此 vibe spec

合約 testid 是業務合約的可定位表面，不應由 vibe 階段自由擴張。
```

## Pattern 模板表

每個 sub-pattern 對應一份模板。Claude 依 hunk 內容填入具體 locator、變數。

### `toggle`（折疊/展開）

```ts
// AUTO-GENERATED by /vibe-e2e
// Source hunk: {path}:{lines}
// Pattern: toggle

import { expect, test } from '@playwright/test'
import { login, resetMockData } from '../helpers'

test.beforeEach(async ({ page }) => {
  await resetMockData(page)
})

test.describe('vibe：{描述：例 sidebar 折疊}', () => {
  test('觸發後目標切換可見性', async ({ page }) => {
    await login(page, 'admin', 'admin888')
    await page.goto('{path}', { waitUntil: 'networkidle' })

    const trigger = page.getByRole('button', { name: /{觸發 name regex}/ })
    const target = page.getByRole('{目標 role}', { name: /{目標 name regex}/ })

    // 初始狀態斷言（visible 或 hidden 由 hunk 推測初值）
    await expect(target).toBeVisible()

    await trigger.click()
    await expect(target).toBeHidden()

    await trigger.click()
    await expect(target).toBeVisible()
  })
})
```

### `tab`

```ts
test('vibe：切換 tab 後對應 panel 出現', async ({ page }) => {
  await login(page, 'admin', 'admin888')
  await page.goto('{path}', { waitUntil: 'networkidle' })

  const targetTab = page.getByRole('tab', { name: /{tab name}/ })
  await targetTab.click()

  const panel = page.getByRole('tabpanel')
  await expect(panel).toContainText(/{預期內容 regex}/)
})
```

### `modal-toggle`

```ts
test('vibe：點觸發開啟 modal、按關閉收起', async ({ page }) => {
  await login(page, 'admin', 'admin888')
  await page.goto('{path}', { waitUntil: 'networkidle' })

  await page.getByRole('button', { name: /{觸發 name}/ }).click()
  const dialog = page.getByRole('dialog')
  await expect(dialog).toBeVisible()

  await dialog.getByRole('button', { name: /關閉|取消|×/ }).click()
  await expect(dialog).toBeHidden()
})
```

### `conditional-render`（一般條件渲染）

```ts
// 若無法從 hunk 推測「怎麼觸發條件」→ skip + TODO
test.skip('vibe：條件渲染 — 待補', async ({ page }) => {
  // TODO: 補上觸發條件的步驟（例：篩選某類別 / 改某個輸入）
  // Source hunk: {path}:{lines}
  // Condition var: {推測的條件變數名}
})
```

### `hover`

```ts
// hover 結果通常是 visual class 變化，自動測意義低 → 預設 skip
test.skip('vibe：hover 效果 — 建議人工視覺確認', async () => {
  // hover 不在 vibe-e2e 自動測範圍
})
```

### `loading-state`

```ts
test('vibe：API 延遲時顯示 loading UI', async ({ page }) => {
  await login(page, 'admin', 'admin888')

  // 攔截目標 API 延遲 1.5 秒
  await page.route('**/api/v1/{endpoint}', async (route) => {
    await new Promise((r) => setTimeout(r, 1500))
    await route.continue()
  })

  await page.goto('{path}')
  // 期望 loading UI 在 API 完成前可見
  await expect(page.getByText(/載入中|loading|skeleton/i)).toBeVisible({ timeout: 1000 })
})
```

### `error-state`

## Locator 推測規則

從 diff hunk 提取：

| 來源 | 推測規則 |
|---|---|
| `<UButton>觸發文字</UButton>` 新增 | `getByRole('button', { name: /觸發文字/ })` |
| `aria-label="X"` 新增 | `getByRole('?', { name: /X/ })` |
| `<UTabs :items="[{ label: 'X' }, ...]">` | `getByRole('tab', { name: /X/ })` |
| `v-model="open"` + `<UModal>` | `getByRole('dialog')` |
| 新 `<section aria-labelledby="X">` | `getByRole('region', { name: /X/ })` |
| 沒有語意 role/name 線索，UI 既有 testid **在白名單內** | `getByTestId('xxx')`（合法，合約一部分） |
| 沒有語意 role/name 線索，UI 既有 testid **不在白名單** | **不可使用**——產 `.skip` + WARNING：「孤兒 testid 候選，請先決定要納入合約還是改用語意 anchor」 |
| 沒有語意 role/name 線索，UI 也沒任何 testid | **不可自己創 `vibe-xxx`**——產 `.skip` + TODO：「此 element 缺合約定位手段，請補 .flow.md 或加語意 anchor」 |

> **與舊版的差別**：舊版 fallback 是「自己創 `vibe-{描述}` testid 並要 UI 補上」——這個路徑會把任意 vibe 階段的 testid 漂白進合約。新版禁止此路徑，未授權 testid 必須先走 flow.md → spec 重生流程。

## Stale 處理

- 每次跑 /vibe-e2e 都會清空 `test/e2e/vibe/*.spec.ts` 再重生（marker 為 AUTO-GENERATED 的才會清）
- 人工想長期維護的測試，請寫到主 spec（`test/e2e/specs/`），不要寫進 vibe 資料夾
- `helpers.ts`（若有）與 `README.md` 不清空

---

## 與相關 skill 的關係

```
/vibe-check    主 spec 守門（先跑、確認綠燈）
   ↓ green
/vibe-setup    分類 + pattern 標記（人類看，純 console）
   ↓ 決策
/vibe-e2e      （這個 skill）依當下 diff 重新分類 + 套版生 spec + 跑
```

`/vibe-e2e` 內部會重跑分類（不依賴 `/vibe-setup` 落地檔，因為前者選了「不落地」設計）。`/vibe-setup` 的角色是給人類預覽決策，不是 `/vibe-e2e` 的前置輸入。

---

## 實作要點

1. **vibe spec 是 auto-gen**：每次重生覆蓋，不要在這裡放人工長期測試
2. **pattern 推測寧可保守**：不確定就 `.skip + TODO`，不要硬生錯誤斷言
3. **locator 推測有層次**：role+name 最佳、testid fallback、不確定就 TODO
4. **失敗回報要可行動**：列出來源 hunk、pattern、嫌疑、建議行動
5. **不污染主 spec**：絕不寫入 `test/e2e/specs/`，不修主 config
